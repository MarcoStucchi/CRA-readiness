## Introduction

As more devices in the home connect to the Internet, the cyber security and data protection of the Internet of Things (IoT) becomes a growing concern. People entrust their personal data to an increasing number of online devices and services. Products and appliances that have traditionally been offline are now connected and need to be designed to withstand cyber threats.

The present document brings together widely considered good practices in security for Internet-connected consumer devices in a set of high-level outcome-focused provisions. The objective is to support all parties involved in the development and manufacturing of consumer IoT with guidance on securing their products.

The provisions are primarily outcome-focused, rather than prescriptive, giving organizations the flexibility to innovate and implement security and data protection solutions appropriate for their products.

The present document is not intended to solve all security, data protection and privacy challenges associated with consumer IoT. It also does not focus on protecting against attacks that are prolonged/sophisticated or that require sustained physical access to the device. Rather, the focus is on the technical controls and organizational policies that matter most in addressing the most significant and widespread security shortcomings.

> **NOTE:** Mappings of the landscape of IoT security standards, recommendations and guidance are available in ENISA Baseline Security Recommendations for IoT – Interactive Tool [i.15] and in Copper Horse Mapping Security & Privacy in the Internet of Things [i.14].

## 1 Scope

The present document specifies high-level security and data protection provisions for consumer IoT devices that are connected to network infrastructure (such as the Internet or home network) and their interactions with associated services.

**Non-exhaustive list of consumer IoT devices:**

- Connected children's toys and baby monitors
- Connected smoke detectors, door locks and window sensors
- IoT gateways, base stations and hubs to which multiple devices connect
- Smart cameras, smart speakers and smart televisions together with their remote controls
- Wearable health trackers
- Connected home automation and alarm systems, especially their gateways and hubs
- Connected appliances, such as washing machines and fridges
- Smart home assistants

The present document also addresses security considerations specific to constraints in device resources (e.g. energy supply, communication bandwidth, processing power or (non-)volatile memory capacity).

Devices not intended for consumer use (e.g. manufacturing, healthcare or other industrial applications) are **not** in scope.

## 3 Definition of Terms, Symbols and Abbreviations

### 3.1 Terms

**administrator** – user who has the highest-privilege level possible for a user of the device, which can mean they are able to change any configuration related to the intended functionality.

**associated services** – digital services that, together with the device, are part of the overall consumer IoT product and that are typically required to provide the product's intended functionality (e.g. mobile applications, cloud computing/storage, third-party APIs).

**authentication mechanism** – method used to prove the authenticity of an entity (user or machine), e.g. password, QR code, biometric fingerprint scanner.

**authentication value** – individual value of an attribute used by an authentication mechanism (e.g. a password character string, a fingerprint).

**best practice** – measures that have been shown to provide appropriate security for the corresponding use case.

**best practice cryptography** – cryptography suitable for the corresponding use case with no indications of a feasible attack using current readily available techniques (covers primitives, implementation, key generation and handling).

**consumer** – natural person acting for purposes outside their trade, business, craft or profession.

**consumer IoT device** – network-connected (and network-connectable) device that has relationships to associated services and is used by the consumer typically in the home or as electronic wearables.

**critical security parameter** – security-related confidential information whose disclosure or modification can compromise the security of the device (e.g. secret cryptographic keys, passwords, PINs, private components of certificates).

**debug interface** – physical interface used by the manufacturer to communicate with the device during development or triage, not used as part of consumer-facing functionality (e.g. test points, UART, SWD, JTAG).

**defined support period** – minimum length of time, expressed as a period or end-date, for which a manufacturer will provide security updates.

**device manufacturer** – entity that creates an assembled final consumer IoT product.

**factory default** – state of the device after factory reset or after final production/assembly.

**initialization** – process that activates the network connectivity of the device for operation and optionally sets authentication features.

**initialized state** – state of the device after initialization.

**IoT product** – consumer IoT device and its associated services.

**isolable** – able to be removed from the network it is connected to with any functionality loss related only to that connectivity; or able to be placed in a self-contained environment with other devices when integrity can be ensured.

**logical interface** – virtual interface used to communicate with the device at a logical layer.

**manufacturer** – relevant economic operator in the supply chain (including the device manufacturer, importers, distributors, integrators, component and platform providers, software providers, etc.).

**network interface** – physical interface that can be used to access the functionality of consumer IoT via a network.

**owner** – user who owns or who purchased the device.

**personal data** – any information relating to an identified or identifiable natural person.

**physical interface** – physical port or air interface (radio, audio or optical) used to communicate with the device at the physical layer (e.g. radios, ethernet ports, USB, debug interfaces).

**public security parameter** – security-related public information whose modification can compromise the security of the device (e.g. public key to verify software update authenticity/integrity, public components of certificates).

**remotely accessible** – intended to be accessible from outside the local network.

**security module** – set of hardware, software, and/or firmware that implements security functions.

**security update** – software update that addresses security vulnerabilities either discovered by or reported to the manufacturer.

**sensitive security parameters** – critical security parameters and public security parameters.

**software service** – software component of a device used to support functionality.

**telemetry data** – data from a device that provides information related to the usage of that device which could help the manufacturer identify security-related issues.

**unique per device** – unique for each individual device of a given product class or type.

**user** – natural person or organization.

**user interface** – user-facing interface of the consumer IoT device (e.g. buttons, screens, speaker, audio/video recorder, or logical frontends).

**vulnerability management process** – course of action in which an organization addresses and remediates a disclosed vulnerability.

## 4 Implementation of the Standard

The implementation of provisions is informed by risk assessment and threat modelling (such as ISO/IEC 27005:2022 [i.27] and STRIDE Threat Model [i.28]) and data protection and privacy impact assessments.

The present document sets a security and data protection baseline. Due to the broad landscape of consumer IoT, the applicability of provisions is dependent on each device. A degree of flexibility is provided through the use of non-mandatory "should" provisions (recommendations).

Testing and certification involving third-party assessment is likely to require documentation including architectural design documentation, security requirements capture, threat models, environmental assumptions, and policy documentation for lifecycle management. A way to assess conformance is specified in ETSI TS 103 701 [i.19].

## 5 Cyber Security Provisions for Consumer IoT

### 5.0 Reporting Implementation

> **Provision 5.0-1 (M):** A justification shall be recorded for each recommendation in the present document that is considered to be not applicable for or not fulfilled by the consumer IoT device.

It is recommended that the manufacturer conducts threat modelling to identify, analyse and mitigate threats to the device. Table B.1 provides a schema to record justifications in a structured manner.

Cases where a provision may be not applicable include:
- Resource constraints that prevent certain security measures
- Functionality described in the provision is not included
- The use case requires compliance with additional industry/regulatory requirements that preclude compliance

---

### 5.1 No Universal Default Passwords

> **Provision 5.1-1 (M):** Where passwords are used to authenticate users against the device or for machine-to-machine authentication, and in any state other than the factory default, all consumer IoT device passwords shall be unique per device or defined by the user.

Many consumer IoT devices are sold with universal default usernames and passwords (e.g. "admin, admin"). Continued usage of universal default values has been the source of many security issues in IoT. This can be achieved by pre-installed unique per-device passwords, by requiring the user to choose a password during initialization, or by other methods that do not use passwords.

> **Provision 5.1-2 (M):** Where pre-installed unique per device passwords are used to authenticate users against the device or for machine-to-machine authentication, these shall be generated with a mechanism that reduces the risk of automated attacks against a class or type of consumer IoT device.

Passwords with incremental counters (e.g. "password1", "password2") are easily guessable. Passwords related in an obvious way to public information (e.g. MAC address or Wi-Fi SSID) can allow for password retrieval using automated means.

> **Provision 5.1-2A (R):** Passwords should not be used for machine-to-machine authentication.

Using human-memorable passwords for machine-to-machine authentication is generally not appropriate, but pre-shared keys can be appropriate when their use follows best practice.

> **Provision 5.1-3 (M):** Authentication mechanisms used to authenticate users against the consumer IoT device or used for machine-to-machine authentication shall use best practice cryptography, appropriate to the properties of the technology, operating environment, risk and usage.

> **Provision 5.1-4 (M):** Where a user can authenticate against a consumer IoT device, the consumer IoT device shall provide to the user or an administrator a simple mechanism to change the authentication value used.

> **Provision 5.1-5 (M, conditional):** The consumer IoT device shall have a mechanism available which makes successful brute-force attacks on authentication mechanisms via network interfaces impracticable unless the device has a resource constraint determined by the use case that prevents the implementation.

*Example: A consumer IoT device has a limitation on the number of authentication attempts within a certain time interval. It also uses increasing time intervals between attempts.*

---

### 5.2 Implement a Means to Manage Reports of Vulnerabilities

Manufacturers that provide IoT products have a duty of care to consumers and third parties who can be harmed by their failure to have a CVD programme in place.

Disclosures can comprise different approaches:
- **Single product/service vulnerabilities**: reported directly to the affected stakeholder (device manufacturer, IoT service provider or mobile application developer).
- **Systemic vulnerabilities**: a problem potentially systemic across products; sharing via a relevant competent industry body can coordinate a wider scale response.

> **Provision 5.2-1 (M):** The manufacturer shall make a vulnerability disclosure policy publicly available. This policy shall include, at a minimum:
> - contact information for the reporting of issues; and
> - timelines for initial acknowledgement of receipt of a vulnerability report; and
> - timelines for when the person who reported the issue will receive status updates until the resolution of the reported issues.

A vulnerability disclosure policy clearly specifies the process through which security researchers and others are able to report issues.

**Coordinated Vulnerability Disclosure (CVD)** is a set of processes for dealing with disclosures about potential security vulnerabilities and to support their remediation. CVD is standardized by ISO in ISO/IEC 29147 [i.4]. **Multi-Party Coordinated Vulnerability Disclosure (MPCVD)** refers to the application of CVD in a multi-stakeholder scenario.

> **Provision 5.2-2 (R):** Disclosed vulnerabilities should be acted on in a timely manner.

Conventionally, the vulnerability management process is completed within 90 days for a software solution, including availability of patches and notification of the issue. A hardware fix can take considerably longer.

> **Provision 5.2-3 (R):** Manufacturers should continually monitor for, identify and rectify security vulnerabilities within consumer IoT products they sell, produce, have produced and associated services they operate during the defined support period.

Creating and maintaining a list of all software components (a Software Bill Of Materials – SBOM) is a pre-requisite to monitor for product vulnerabilities.

---

### 5.3 Keep Software Updated

Developing and deploying security updates in a timely manner is one of the most important actions a manufacturer can take to protect its customers.

It is often advisable not to bundle security updates with more complex software updates such as feature updates.

> **Provision 5.3-1 (R):** All software components in consumer IoT devices that are not immutable due to security reasons should be securely updateable.

> **Provision 5.3-2 (M, conditional):** The consumer IoT device shall have a secure update mechanism unless the device has a resource constraint determined by the use case that prevents the implementation.

"Securely updateable" and "secure update mechanism" means that there are adequate measures to prevent an attacker misusing the update mechanism (e.g. use of authentic software update servers, integrity-protected communications channels, verifying the authenticity and integrity of software updates; anti-rollback policy based on version checking).

> **Provision 5.3-3 (M):** An update shall be simple for the user to apply.

> **Provision 5.3-4A (R):** One secure update mechanism should be configurable to be automated.

> **Provision 5.3-4B (R):** During initialization the consumer IoT device should activate an automatic secure update mechanism after user's consent.

> **Provision 5.3-5 (R):** The consumer IoT device or an associated service should check after initialization whether security updates are available.

> **Provision 5.3-6A (R):** If the consumer IoT device supports automated updates, the user should be able to enable and disable the automatic update mechanism and postpone the installation of updates provided via an automatic update mechanism.

> **Provision 5.3-6B (R):** If the consumer IoT device supports update notifications, the user should be able to enable and disable the update notifications.

> **Provision 5.3-7 (M):** The consumer IoT device shall use best practice cryptography to facilitate secure update mechanisms.

> **Provision 5.3-8 (M, conditional):** Security updates shall be timely.

> **Provision 5.3-9 (R):** The consumer IoT device should verify the authenticity and integrity of software updates.

> **Provision 5.3-10 (M):** Where updates are delivered over a network interface, the consumer IoT device shall verify the authenticity and integrity of each update via a trust relationship.

*Valid trust relationships include: authenticated communication channels, presence on a network requiring a critical security parameter to join, digital signature-based verification, or confirmation by the user.*

> **Provision 5.3-11 (R, conditional):** The user should be informed in a recognizable and apparent manner that a security update is required together with information on the risks mitigated by that update.

> **Provision 5.3-12 (R, conditional):** The user should be notified when the application of a software update will disrupt the basic functioning of the consumer IoT device.

*Example: During an update, a watch will continue to display the time, a home thermostat will continue to maintain a reasonable temperature and a Smart Lock will continue to lock and unlock a door.*

> **Provision 5.3-13 (M):** The manufacturer shall publish, in an accessible way that is clear and transparent to the user, the defined support period.

> **Provision 5.3-14 (R, conditional):** For consumer IoT devices that cannot have their software updated, the rationale for the absence of software updates, the period and method of hardware replacement support should be published by the manufacturer in an accessible way that is clear and transparent to the user.

> **Provision 5.3-15A (R, conditional):** For devices that cannot have their software updated, the consumer IoT device should be isolable.

> **Provision 5.3-15B (R, conditional):** For devices that cannot have their software updated, the consumer IoT device hardware should be replaceable.

> **Provision 5.3-16 (M):** The model designation of the consumer IoT device shall be clearly recognizable, either by labelling on the consumer IoT device or via a user interface.

### 5.4 Securely Store Sensitive Security Parameters

> **Provision 5.4-1 (M):** Sensitive security parameters in persistent storage shall be stored securely by the consumer IoT device.

Stored critical security parameters need adequate measures to prevent both modification and disclosure; stored public security parameters only need adequate measures to prevent modification. Appropriate mechanisms include those provided by a TEE, encrypted storage associated with the hardware, Secure Elements (SE) or Dedicated Security Components (DSC), and processing capabilities of software running on a UICC.

> **Provision 5.4-2 (M):** Where a hard-coded unique per device identity is used in a consumer IoT device for security purposes, it shall be implemented in such a way that it resists tampering by means such as physical, electrical or software.

> **Provision 5.4-3 (M):** Hard-coded critical security parameters in consumer IoT device software source code shall not be used.

Reverse engineering of consumer IoT devices and applications can easily discover credentials such as hard-coded usernames and passwords in software. Simple obfuscation methods can be trivially broken.

> **Provision 5.4-4 (M):** Any critical security parameters used for integrity and authenticity checks of software updates and for protection of communication with associated services in consumer IoT device software shall be unique per device and shall be produced with a mechanism that reduces the risk of automated attacks against classes of consumer IoT devices.

---

### 5.5 Communicate Securely

> **Provision 5.5-1 (M):** The consumer IoT device shall use best practice cryptography to communicate securely.

> **Provision 5.5-2 (R):** The consumer IoT device should use reviewed or evaluated implementations to deliver network and security functionalities, particularly in the field of cryptography.

> **Provision 5.5-3 (R):** Cryptographic algorithms and primitives should be replaceable.

*This is a building block for "cryptoagility". Note: changing cryptographic algorithms and primitives in configuration is a security-relevant change.*

> **Provision 5.5-4 (R):** Access to consumer IoT device functionality via a network interface in the initialized state should only be possible after authentication on that interface.

> **Provision 5.5-5 (M):** Consumer IoT device functionality that allows security-relevant changes in configuration via a network interface shall only be accessible after authentication. The exception is for network service protocols that are relied upon by the consumer IoT device and where the manufacturer cannot guarantee what configuration will be required.

*Protocols that are an exception include ARP, DHCP, DNS, ICMP, and NTP. Security-relevant changes include permission management, configuration of network keys and password changes.*

> **Provision 5.5-6 (R):** Critical security parameters should be encrypted in transit, with such encryption appropriate to the properties of the technology, risk and usage.

> **Provision 5.5-7 (M):** The consumer IoT device shall protect the confidentiality of critical security parameters that are communicated via remotely accessible network interfaces.

> **Provision 5.5-8 (M, conditional):** The manufacturer shall follow secure management processes for critical security parameters that relate to the consumer IoT device and for the entire lifecycle of the critical security parameters.

### 5.6 Minimize Exposed Attack Surfaces

The "principle of least privilege" is a foundation stone of good security engineering.

> **Provision 5.6-1 (M):** All unused network interfaces and all unused logical interfaces that are accessible through a network interface shall be disabled.

> **Provision 5.6-2 (M):** In the initialized state, the network interfaces of the consumer IoT device shall minimize the unauthenticated disclosure of security-relevant information.

*Example: security-relevant information could include device configuration, kernel version or software version.*

> **Provision 5.6-3 (R):** Consumer IoT device hardware should not unnecessarily expose physical interfaces to attack.

> **Provision 5.6-4A (M):** Debug interfaces shall be disabled or protected via a best practice authentication or access control mechanism.

Debug interfaces are to be disabled in software by default. It must be impossible for the user to enable any disabled debug interface that is not considered to be required for the consumer IoT device's lifecycle.

> **Provision 5.6-4B (R):** Debug interfaces that are physical ports should be physically protected by the device.

*Note: If a debug interface has been disabled, it is not necessary to physically protect it.*

> **Provision 5.6-5 (R):** The manufacturer should only enable software services that are used or required for the intended use or operation of the consumer IoT device.

> **Provision 5.6-6 (R):** Code should be minimized to the functionality necessary for the consumer IoT device to operate.

> **Provision 5.6-7 (R):** Software should run with least necessary privileges, taking account of both security and functionality.

*Example: Minimal daemons/processes run with "root" privileges. Processes that use network interfaces require unprivileged users.*

> **Provision 5.6-8 (R):** The consumer IoT device should include a hardware-level access control mechanism for memory.

Suitable mechanisms include MMUs or MPUs, executable space protection (e.g. NX bits), memory tagging, and trusted execution environments.

> **Provision 5.6-9 (R):** The manufacturer should follow secure development processes for software deployed on the consumer IoT device.

---

### 5.7 Ensure Software Integrity

> **Provision 5.7-1 (R):** The consumer IoT device should verify its software using secure boot mechanisms.

A hardware root of trust is one way to provide strong attestation as part of a secure boot mechanism. A hardware root of trust is a component from which all other components derive their "trust" – the source of cryptographic trust within that system.

> **Provision 5.7-2 (R):** If an unauthorized change is detected to the software, the consumer IoT device should alert the user and/or administrator to the issue and should not connect to wider networks than those necessary to perform the alerting function.

---

### 5.8 Ensure That Personal Data is Secure

> **Provision 5.8-1 (R):** The confidentiality of personal data transiting between the consumer IoT device and associated services should be protected with best practice cryptography, appropriate to the properties of the technology, operating environment, risk and usage.

> **Provision 5.8-2 (M):** The confidentiality of sensitive personal data communicated between the consumer IoT device and associated services shall be protected with best practice cryptography, appropriate to the properties of the technology, operating environment, risk and usage.

*"Sensitive personal data" examples: video stream of a home security camera, payment information, content of communication data, and timestamped location data.*

> **Provision 5.8-3 (M):** All external sensing capabilities of the consumer IoT device shall be documented in an accessible way that is clear and transparent for the user.

*Example: An external sensing capability can be an optic or acoustic sensor.*

---

### 5.9 Make Systems Resilient to Outages

> **Provision 5.9-1 (R):** Resilience should be built in to consumer IoT products, taking into account the possibility of outages of data networks and power.

> **Provision 5.9-2 (R):** Consumer IoT devices should remain operating and locally functional in the case of a loss of network access and should recover cleanly in the case of restoration of a loss of power.

*"Recovering cleanly" normally involves resuming connectivity and functionality in the same or improved state.*

> **Provision 5.9-3 (R):** The consumer IoT device should connect to networks in an expected, operational and stable state and in an orderly fashion, taking the capability of the infrastructure into consideration.

*Orderly reconnection means taking explicit steps to avoid simultaneous requests (e.g. introducing a random delay before a reconnection attempt according to an incremental back-off mechanism).*

---

### 5.10 Examine System Telemetry Data

> **Provision 5.10-1 (R):** If telemetry data is collected from consumer IoT products, such as usage and measurement data, it should be examined for security anomalies.

*Security anomalies can be represented by a deviation from normal behaviour, for example an abnormal increase of failed login attempts.*

Examining telemetry, including log data, is useful for security evaluation and allows unusual circumstances to be identified early.

---

### 5.11 Make It Easy for Users to Delete User Data

The term "delete" refers to the software-based process of making data no longer present, accessible or recognizable. "Erasure" is a deliberate process with the aim of making it impractical to recover the data (e.g. by overwriting).

> **Provision 5.11-1 (M):** Users shall be provided with functionality such that all their user data can be erased from the consumer IoT device in a simple manner.

*"User data" means all individual data stored on the IoT device including personal data, user configuration and cryptographic material such as user passwords or keys.*

> **Provision 5.11-2 (R):** The consumer should be provided with functionality on the consumer IoT device such that personal data can be deleted from associated services in a simple manner.

> **Provision 5.11-3 (R):** Users should be given clear instructions on how to delete and where possible to erase their personal data from the device and associated services.

> **Provision 5.11-4 (R):** Users should be provided with clear confirmation that personal data has been deleted and where possible erased from devices and associated services.

---

### 5.12 Make Installation and Maintenance of Devices Easy

> **Provision 5.12-1 (R):** Installation and maintenance of consumer IoT should involve minimal decisions by the user and should follow security best practice on usability.

*Example: The user uses a wizard to set up the consumer IoT device where a subset of configuration options is presented with common defaults already specified and appropriate security options already turned on by default.*

> **Provision 5.12-2 (R):** The manufacturer should provide users with guidance on how to securely set up their consumer IoT device.

> **Provision 5.12-3 (R):** The manufacturer should provide users with guidance on how to check whether their consumer IoT device is securely set up and maintained in a secure state.

---

### 5.13 Validate Input Data

> **Provision 5.13-1A (M):** Data input at application layer to the device via user interfaces shall be validated by the device regarding unexpected data input to prevent system manipulations and failures.

*Example: Manipulation of the integrated database via SQL injection over a user interface is mitigated via discarding invalid data input.*

> **Provision 5.13-1B (M):** Data input at application layer to the device via network interfaces shall be validated by the device regarding unexpected data input to prevent system manipulations and failures.

*This includes network accessible APIs without prior authentication and network accessible APIs for authentication mechanisms.*

## 6 Data Protection Provisions for Consumer IoT

Many consumer IoT devices process personal data. Consumers have an expectation that they will be able to protect their personal data by configuring IoT device and associated service functionality appropriately. There also exist laws and regulations relating to the protection of personal data in consumer IoT devices (e.g. GDPR [i.7]).

> **Provision 6-1 (M):** The manufacturer shall provide consumers with clear and transparent information about what personal data is processed and for what purposes, by whom, and for how long, for each consumer IoT device and associated service. This also applies to third parties that can be involved, including advertisers.

> **Provision 6-2 (M):** Where personal data will be processed on the basis of consumers' consent, the consumer IoT device shall provide a means to acquire this consent in a valid way.

Obtaining consent "in a valid way" normally involves giving consumers a free, obvious and explicit opt-in choice.

> **Provision 6-3A (M):** Where personal data will be processed on the basis of consumers' consent, the consumer IoT device shall provide a means to withdraw this consent at any time.

> **Provision 6-3B (M):** Where personal data will be processed on the basis of consumers' consent, the consumer IoT device shall provide a means of storing information about this consent.

*Example: The device stores timestamps indicating when consent was given or withdrawn, and the data processing purposes covered by the consent.*

> **Provision 6-4 (R):** If telemetry data is collected from consumer IoT products, the processing of personal data should be limited to that which is necessary for the intended functionality identified in provision 6-5.

> **Provision 6-5 (M):** If telemetry data is collected from consumer IoT products, consumers shall be provided with information on what telemetry data is collected, how it is being used, by whom, and for what purposes.

> **Provision 6-6 (M):** Data stored and processed on a consumer IoT device, or made available to an associated service by the consumer IoT device, for purposes identified in provision 6-1 shall be limited to that which is necessary for the purpose for which it is being collected or processed, and deleted once no longer necessary for any of the purposes identified.

> **Provision 6-7 (R):** When the purpose of data collection from consumer IoT devices, or processing on the consumer IoT device, is solely to compute an aggregate result, the data collected should be the minimum required to compute the aggregate, the aggregation should happen as early as possible, and the retention of both collected data and the resulting aggregate should be minimized.

*Example: Federated learning enables multiple devices to collaboratively train machine learning models under coordination of a central server. Each device's raw data is stored locally and not exchanged; only focused updates are uploaded.*

> **Provision 6-8 (R):** Data anonymization technologies should be used to protect privacy during data collection, processing and storage.

## Annex A (informative): Basic Concepts and Models

### A.1 Architecture

A consumer IoT device is a collection of hardware and software components, generally with physical interfaces which can also be network interfaces.

Consumer IoT deployed in the home will often consist of a variety of consumer IoT devices connected to the LAN, either directly through IP connectivity (Ethernet or Wi-Fi), or indirectly via a gateway or hub using non-IP connectivity (e.g. protocols based on IEEE 802.15.4). A router then connects the LAN to the WAN (the Internet).

**Associated services** are those included by the manufacturer (e.g. telemetry, companion mobile application) or installed as part of initialization. Services the user chooses to install independently are **not** counted as associated services.

Examples:
- Websites accessed via a consumer IoT device's browser → **not** associated services
- Apps installed by default that the user cannot easily delete → generally **associated services**
- Apps installed through a store at the choice of the user → **not** associated services
- Connecting to a telemetry platform → **associated service** (pre-configured by the device manufacturer)

### A.2 Device States

A consumer IoT device (from manufacturing to decommissioning) will transition between several states:

| State | Description |
|-------|-------------|
| **Factory Default** | After factory reset or final production/assembly. Only network interfaces required for initialization usable. Unique Factory Default authentication value (if present) is used. No personal data or custom configuration present. |
| **Initialized (without personal data)** | Authentication mechanism selected by user, if changed. Network interfaces required for functionality selected by user are usable. |
| **Initialized (with personal data)** | Personal data present on the device. |
| **Initialized (with personal data and additional users)** | Personal data for additional users present. Additional accounts potentially created. |

A decommissioned consumer IoT device would be in the Factory Default state, as the Factory Reset process is used to remove all user data and configuration.

### A.3 Interfaces

Types of interfaces defined and used in the present document:

- **Physical interfaces**: antennas or RJ45 sockets for network communication; serial interfaces for debugging (JTAG, UART); wired point-to-point connections (USB). Enable communication at the physical layer.
- **Logical interfaces**: web interfaces for users, APIs for other devices or services, or provided by network stacks. Typically make use of physical interfaces for communication.
- **Network interfaces**: physical interfaces that can be used to access functionality via a network (e.g. Wi-Fi, Ethernet).
- **Debug interfaces**: physical interfaces used during development/triage (e.g. JTAG).
- **User interfaces**: enable direct interaction with the user (buttons, screens, speaker, audio/video recorder) or implemented as logical interfaces.
