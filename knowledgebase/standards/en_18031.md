# EN 18031 Series — Common Security Requirements for Radio Equipment

> **Mandatory from 1 August 2025** under the Radio Equipment Directive (RED) 2014/53/EU for all
> internet-connected radio equipment placed on the EU market.
> Harmonised in the Official Journal of the EU on 30 January 2025 (OJ L 2025/138).
>
> **Relationship to CRA:** EN 18031 is the current mandatory cybersecurity baseline for wireless
> consumer products under RED. It serves as a stepping stone toward CRA compliance (full CRA
> applicability: 11 December 2027). Compliance work done for EN 18031 directly reduces the effort
> needed for CRA conformity assessment.
>
> **Do not confuse with ISO/IEC 18031:2025** (random bit generation — entirely unrelated).

---

## Overview: Three-part structure

| Standard | Title | Scope | RED Article |
|---|---|---|---|
| **EN 18031-1:2024** | Common security requirements — Part 1: Internet connected radio equipment | All radio equipment that connects to the internet (directly or indirectly) | Art. 3(3)(d) — network protection |
| **EN 18031-2:2024** | Common security requirements — Part 2: Radio equipment processing data | Internet-connected + childcare + toys + wearable radio equipment that processes personal/location data | Art. 3(3)(e) — privacy and data protection |
| **EN 18031-3:2024** | Common security requirements — Part 3: Radio equipment processing virtual money or monetary value | Internet-connected radio equipment handling virtual currencies, payments, or monetary values | Art. 3(3)(f) — fraud prevention |

For a typical connected consumer thermostat: **EN 18031-1** applies (internet-connected radio equipment) and **EN 18031-2** applies (processes personal data — occupancy, scheduling, user accounts). EN 18031-3 does not apply unless the device handles payments.

## EN 18031-1 — Internet connected radio equipment

### Scope

Specifies common security requirements and related assessment criteria for internet-connected radio equipment. The standard defines a baseline of security mechanisms that equipment must implement to protect its security assets and network assets against common cybersecurity threats and to mitigate publicly known exploitable vulnerabilities.

Defence in depth is an explicit design principle: no single measure suffices; achieving security objectives requires a coordinated suite of mechanisms.

### Requirements structure

Each requirement in EN 18031-1 follows a consistent pattern:
- **XXX-1**: Applicability — determine whether the mechanism is required for this equipment
- **XXX-2**: Appropriateness — if required, determine whether it is implemented correctly
- Each clause includes: Requirement, Rationale, Guidance, Assessment criteria (conceptual + functional completeness + functional sufficiency assessments)

### Requirement clauses

| Clause | ID | Title | Summary |
|---|---|---|---|
| 6.1 | ACM | Access control mechanism | Equipment must implement access control for security assets and network assets accessible by entities via network or user interfaces |
| 6.1.1 | ACM-1 | Applicability of access control mechanisms | For each security asset and network asset, determine whether access control is required via decision tree |
| 6.1.2 | ACM-2 | Appropriate access control mechanisms | Access control must ensure only authorised entities have access; supports RBAC, DAC, MAC, and generic models |
| 6.2 | AUM | Authentication mechanism | Equipment must use authentication for access control via network and user interfaces to confidential/sensitive assets |
| 6.2.1 | AUM-1 | Applicability of authentication mechanisms | Authentication required for network interfaces (AUM-1-1) and user interfaces (AUM-1-2) giving access to confidential/sensitive assets |
| 6.2.2 | AUM-2 | Appropriate authentication mechanisms | Authentication mechanism must be appropriate to the risk and asset type |
| 6.2.3 | AUM-3 | Authenticator validation | Authenticators must be validated before granting access |
| 6.2.4 | AUM-4 | Changing authenticators | Equipment must support changing authenticators; authorisation required |
| 6.2.5 | AUM-5 | Password strength | Password-based authenticators must meet minimum strength requirements; default passwords must be unique per device or changeable |
| 6.2.6 | AUM-6 | Brute force protection | Equipment must implement protection against brute force attacks on authenticators |
| 6.3 | SUM | Secure update mechanism | Equipment must implement a secure update mechanism if software updates are supported |
| 6.3.1 | SUM-1 | Applicability of update mechanisms | Determine if update mechanisms are required |
| 6.3.2 | SUM-2 | Secure updates | Updates must be authenticated and integrity-protected; device must verify before installing |
| 6.3.3 | SUM-3 | Automated updates | If automated updates are supported, mechanism must operate securely |
| 6.4 | SSM | Secure storage mechanism | Equipment must protect security assets stored persistently |
| 6.4.1 | SSM-1 | Applicability of secure storage | Determine which assets require secure storage |
| 6.4.2 | SSM-2 | Integrity protection for secure storage | Stored security assets must be integrity-protected |
| 6.4.3 | SSM-3 | Confidentiality protection for secure storage | Confidential security assets must be protected from disclosure |
| 6.5 | SCM | Secure communication mechanism | Equipment must protect communications carrying security assets or network assets |
| 6.5.1 | SCM-1 | Applicability of secure communication | Determine which communications require protection |
| 6.5.2 | SCM-2 | Integrity and authenticity protection | Communications must be integrity and authenticity protected |
| 6.5.3 | SCM-3 | Confidentiality protection | Confidential communications must be encrypted |
| 6.5.4 | SCM-4 | Replay protection | Communications must be protected against replay attacks where applicable |
| 6.6 | RLM | Resilience mechanism | Equipment must implement resilience mechanisms to maintain essential functions |
| 6.6.1 | RLM-1 | Applicability and appropriateness of resilience mechanisms | Determine and assess resilience requirements for essential functions |
| 6.10 | GEC | General equipment capabilities | Baseline security hygiene requirements applicable to all equipment |
| 6.10.1 | GEC-1 | No publicly known exploitable vulnerabilities | Equipment must not contain publicly known exploitable vulnerabilities at time of delivery |
| 6.10.2 | GEC-2 | Limit exposure of services | Services exposed via network interfaces must be limited to what is necessary |
| 6.10.3 | GEC-3 | Configuration of optional services | Optional services exposed via network interfaces must be configurable (enable/disable) |
| 6.10.4 | GEC-4 | Documentation of exposed interfaces | Manufacturer must document all exposed network interfaces and services |
| 6.10.5 | GEC-5 | No unnecessary external interfaces | Equipment must not expose unnecessary external interfaces |
| 6.10.6 | GEC-6 | Input validation | Equipment must validate inputs received via network and user interfaces |
| 6.11 | CRY | Cryptography | All cryptography used must follow best practice |
| 6.11.1 | CRY-1 | Best practice cryptography | Cryptographic mechanisms must be appropriate for the protection goals; reference: SOGIS Agreed Cryptographic Mechanisms |

## EN 18031-2 — Radio equipment processing data

### Scope

Extends EN 18031-1 for radio equipment that processes personal data, location data, or data related to children. Covers: internet-connected radio equipment, childcare radio equipment, toy radio equipment, and wearable radio equipment. Addresses RED Article 3(3)(e) — protection of privacy and personal data.

### Additional requirements vs EN 18031-1

EN 18031-2 inherits all clauses from EN 18031-1 and adds:

| Clause | ID | Title | Summary |
|---|---|---|---|
| 6.1.3 | ACM-3 | Default access control for children in toys | Toys must implement default access control protecting children's data from third parties |
| 6.1.4 | ACM-4 | Default access control to children's privacy assets | Toys and childcare equipment must protect children's privacy assets from third-party access by default |
| 6.1.5 | ACM-5 | Parental/Guardian access controls in toys | Toys must provide parental/guardian controls over children's data access |
| 6.1.6 | ACM-6 | Parental/Guardian access for managed assets | Parental controls must extend to all entities' access to managed assets in toys |
| 6.6.1 | LGM-1 | Applicability of logging mechanisms | Determine whether logging mechanisms are required |
| 6.6.2 | LGM-2 | Persistent storage of log data | Log data must be stored persistently where required |
| 6.6.3 | LGM-3 | Minimum number of persistently stored events | Minimum event types that must be logged (access, authentication failures, configuration changes) |
| 6.6.4 | LGM-4 | Time-related information in logs | Log entries must include accurate time information |
| 6.7.1 | DLM-1 | Applicability of deletion mechanisms | Equipment must implement mechanisms to delete personal data when required |
| 6.8.1 | UNM-1 | Applicability of user notification mechanisms | Determine whether user notification is required |
| 6.8.2 | UNM-2 | Appropriate user notification content | User notifications must contain appropriate information about security-relevant events |
| 6.9.1 | CCK-1 | Appropriate confidential cryptographic keys | Confidential cryptographic keys must be appropriate for their purpose |
| 6.9.2 | CCK-2 | CCK generation mechanisms | Confidential cryptographic keys must be generated using appropriate mechanisms |
| 6.9.3 | CCK-3 | Preventing static default values for preinstalled CCKs | Pre-installed confidential cryptographic keys must not use static default values across device population |
| 6.10.7 | GEC-7 | Documentation of external sensing capabilities | Manufacturer must document all external sensing capabilities (cameras, microphones, occupancy sensors, etc.) |

## EN 18031-3 — Radio equipment processing virtual money or monetary value

### Scope

Extends EN 18031-1 for radio equipment that processes virtual currencies, cryptocurrencies, or monetary values. Addresses RED Article 3(3)(f) — prevention of fraudulent access.

### Requirements vs EN 18031-1

EN 18031-3 inherits all EN 18031-1 clauses and adds financial/transaction-specific requirements:

| Clause | ID | Title | Summary |
|---|---|---|---|
| 6.6.1–6.6.4 | LGM | Logging mechanism | Same logging requirements as EN 18031-2 — mandatory for transaction audit trail |
| 6.7.1–6.7.3 | CCK | Confidential cryptographic keys | Extended CCK requirements for financial-grade cryptographic key management |
| 6.8.7 | GEC-7 | *(equipment-specific)* | Additional general equipment capability requirements for monetary value processing |
| 6.8.8 | GEC-8 | Equipment integrity | Equipment handling monetary values must implement integrity protection mechanisms |

## Mapping to ETSI EN 303 645

Each EN 18031-x standard includes an informative Annex C mapping its requirements to the corresponding EN 303 645 provisions. This enables manufacturers who have already assessed against EN 303 645 to identify the delta work needed for EN 18031 compliance.

Key mapping relationships for EN 18031-1 ↔ EN 303 645:

| EN 18031-1 clause | EN 303 645 provision | Topic |
|---|---|---|
| ACM-1, ACM-2 | 5.5-5, 5.5-6, 5.5-7 | Access control to management interfaces |
| AUM-1, AUM-2 | 5.1-1, 5.1-3, 5.1-4 | Authentication; unique per-device credentials |
| AUM-5 | 5.1-1, 5.1-2 | Password strength; no universal default passwords |
| AUM-6 | 5.1-3 | Brute force protection |
| SUM-1, SUM-2 | 5.3-2, 5.3-3, 5.3-10 | Secure update mechanism; update integrity verification |
| SUM-3 | 5.3-3 | Automated updates |
| SSM-1, SSM-2, SSM-3 | 5.4-1, 5.4-2, 5.4-3 | Secure storage; credential and key protection |
| SCM-1, SCM-2, SCM-3 | 5.5-1, 5.5-2, 5.5-3 | Secure communication; TLS; encryption of data in transit |
| GEC-1 | 5.3-2, 5.7-1 | No known exploitable vulnerabilities; vulnerability monitoring |
| GEC-2, GEC-5 | 5.6-1, 5.6-2, 5.6-3 | Minimal attack surface; no unnecessary network services |
| GEC-6 | 5.13-1A, 5.13-1B | Input validation |
| CRY-1 | 5.5-1 | Best practice cryptography |

> **Important for Path A gap analysis:** EN 303 645 and EN 18031-1 share significant overlap but are
> not equivalent. EN 18031-1 is more prescriptive and includes formal assessment criteria (conceptual
> + functional completeness + functional sufficiency assessments) not present in EN 303 645.
> EN 303 645 compliance does **not** automatically imply EN 18031-1 compliance — delta assessment
> is required, particularly for RLM (resilience) which has no direct EN 303 645 equivalent.

## Conformity assessment

| Standard | Assessment route | Presumption of conformity |
|---|---|---|
| EN 18031-1/2/3 | Manufacturer self-assessment + Declaration of Conformity (DoC) under RED | Yes — when applied without restricted clauses |
| Restricted clauses | Notified body involvement required | See OJ L 2025/138 for full restrictions list |

> **Key restriction (all three parts):** Clauses 6.2.5.1 and 6.2.5.2 (AUM-5 — default passwords):
> these clauses permit manufacturers to allow users not to set/use a password. The EU Commission
> has determined that if this option is implemented, the relevant authentication risks are not properly
> addressed and conformity with RED Article 3(3)(d)(e)(f) cannot be presumed. A notified body
> must be consulted if this implementation option is chosen.

## Timeline and regulatory context

| Date | Event |
|---|---|
| November 2024 | EN 18031-1/2/3:2024 published |
| 30 January 2025 | Harmonised in OJ L 2025/138; presumption of conformity granted (with restrictions) |
| **1 August 2025** | **Mandatory for all new internet-connected radio equipment placed on EU market** |
| 11 December 2027 | CRA full applicability — EN 18031 compliance work directly supports CRA preparation |

Products placed on the market before 1 August 2025 and not subsequently substantially modified are not required to comply retroactively, but all new production after that date must comply.

## Applicability to product archetypes

| Product type | EN 18031-1 | EN 18031-2 | EN 18031-3 |
|---|---|---|---|
| Connected thermostat (Wi-Fi) | ✅ Always | ✅ Processes occupancy/schedule/user data | ❌ No monetary value |
| Smart speaker | ✅ Always | ✅ Processes voice/personal data | ❌ Unless used for payments |
| Connected camera / baby monitor | ✅ Always | ✅ Processes video/personal data | ❌ |
| Wearable fitness tracker (BLE+Wi-Fi) | ✅ Always | ✅ Processes health/location data | ❌ Unless payment-enabled |
| Smart payment terminal | ✅ Always | ✅ May process personal data | ✅ Handles monetary value |
| Industrial IoT gateway | Depends — if radio-equipped and internet-connected | Depends on data processed | ❌ |

## Role in the assessment system

| Path | Role of EN 18031 |
|---|---|
| **Path A — Compliance mapping** | EN 18031-1 (and -2 where applicable) clauses are the evaluation criteria for internet-connected radio products, complementing EN 303 645. The CRA_mapping_reconciliation.md delta analysis should be extended to cover EN 18031 vs CRA Annex I when assess_path_a.py is built. |
| **Path B — Threat model** | EN 18031 requirement IDs (e.g. ACM-1, AUM-5, GEC-2) can be referenced in threat hints — a threat identified against a component maps to specific EN 18031 requirements that address it. |
| **Path C — Elicitation** | Archetype `applicable_standards` should include `"EN 18031-1:2024"` and `"EN 18031-2:2024"` for all consumer Wi-Fi products. Elicitation questions should cover RLM (resilience) and GEC-7 (sensing documentation) which have no EN 303 645 equivalent. |
